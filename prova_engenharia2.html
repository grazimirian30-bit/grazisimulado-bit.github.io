Questão 1

A Engenharia de Requisitos é o processo de descobrir, analisar, documentar e verificar os requisitos de um determinado sistema...

Alternativas:

a) Na fase de Especificação...

b) Na etapa de levantamento de requisitos, os desenvolvedores trabalham com os usuários finais para descobrir informações sobre o domínio da aplicação e dos serviços, sem se preocupar com desempenho e restrições de hardware

c) A engenharia de requisitos abrange...

d) A engenharia de requisitos se inicia com os programadores...

e) No documento gerado pela engenharia de requisitos...

✅ Gabarito: alternativa B

O levantamento de requisitos foca na descoberta das necessidades junto aos usuários. O foco inicial não é desempenho ou hardware, mas entender o domínio do problema.

Análise direta das alternativas

❌ Opção 1 — Errada Fala que na especificação já existem layouts, gráficos e codificação prontos para uso. 👉 Especificação descreve requisitos. Não é fase de implementar nem deixar pronto para uso.

✅ Opção 2 — Correta Na elicitação/levantamento, o foco é entender o problema com os usuários. 👉 Detalhes técnicos profundos (como otimização de hardware) não são a prioridade inicial — primeiro vem o entendimento do domínio e das necessidades.

❌ Opção 3 — Errada Inclui codificação e testes dentro da engenharia de requisitos.

❌ Opção 4 — Errada Diz que começa com programadores. 👉 Começa com stakeholders (clientes, usuários, etc.).

❌ Opção 5 — Errada Descreve estudo de viabilidade, não engenharia de requisitos.

✅ Resposta final: alternativa 2.

Boa, bora blindar isso na tua cabeça 🔒✨

🧠 Macete ninja da Engenharia de Requisitos

Pensa assim:

Primeiro ENTENDER → depois DESCREVER → depois VALIDAR → depois GERENCIAR

Em ordem clássica:

Concepção – ideia inicial do sistema

Levantamento (elicitação) – conversar com usuários 👥

Elaboração/análise – organizar e refinar

Negociação – alinhar conflitos

Especificação – documentar tudo 📝

Validação – conferir se está correto

Gerenciamento – controlar mudanças 🔄

🚨 Pegadinhas que caem MUITO em prova

Grava isso:

❌ Não tem codificação na engenharia de requisitos

❌ Não tem testes de software (no sentido de execução do sistema)

❌ Não começa pelos programadores

❌ Especificação não implementa nada

🎯 Frase pra decorar (nível prova)

Levantamento = conversar e entender o usuário

Se a alternativa falar algo tipo:

“descobrir necessidades”

“entender o domínio”

“trabalhar com usuários”

👉 Quase sempre é a correta.

Questão 2

As declarações de serviços que o sistema deve fornecer, de como o sistema deve reagir a entradas específicas...

Alternativas:

a) os requisitos de produto também podem explicar o que o sistema não deve fazer.

b) os requisitos não funcionais também podem explicar o que o sistema não deve fazer.

c) os requisitos de domínio também podem explicar o que o sistema não deve fazer.

d) os requisitos de software também podem explicar o que o sistema não deve fazer.

e) os requisitos funcionais também podem explicar o que o sistema não deve fazer.

✅ Gabarito corrigido: alternativa B (não funcionais)

Embora a descrição inicial seja de requisitos funcionais, a formulação clássica indica que, em alguns casos, requisitos não funcionais também podem declarar o que o sistema não deve fazer.

Aqui tem pegadinha conceitual…

O enunciado descreve requisitos funcionais, mas a frase clássica do Sommerville é:

“Em alguns casos, requisitos não funcionais também podem explicar o que o sistema não deve fazer.”

📌 Ou seja:

A definição principal é funcional

Mas o trecho final da frase clássica aponta para não funcionais

👉 Minha resposta anterior foi a que tem maior chance de estar errada.

Gabarito mais aceito em provas:

✅ os requisitos não funcionais também podem explicar o que o sistema não deve fazer.

Questão 3

Técnica de levantamento que apresenta uma forma rápida de se obter dados quando há muitos usuários...

Alternativas:

a) Entrevista

b) Brainstorming

c) Questionário

d) VORD

e) Workshop

✅ Gabarito: alternativa C (Questionário)

Questionários são ideais quando há muitos usuários distribuídos geograficamente e é necessário coletar dados padronizados rapidamente.

🧠 Por quê?

O enunciado descreve uma técnica que:

coleta dados de muitos usuários 👥

usuários estão em locais diferentes 🌍

precisa ser rápida ⚡

mesma informação para todo mundo

👉 Isso é a cara de questionário (survey).

Questionário é usado justamente quando você precisa escala + padronização + distância.

🚫 Por que as outras não são

❌ Entrevista → é individual e demorada

❌ Brainstorming → geração de ideias em grupo

❌ VORD → técnica de análise orientada a pontos de vista

❌ Workshop → reunião presencial com stakeholders

✅ Gabarito: Questionário

Questão 5

Classifique V ou F sobre tipos de requisitos...

✅ Gabarito: V – V – V – V – F

Requisitos de domínio podem gerar novos requisitos funcionais, portanto a última afirmação é falsa.

Vamos checar uma por uma, rapidinho e com precisão 👇

✅ (1) Verdadeiro

Requisitos do usuário… compreensíveis… podem ser em língua natural.

✔️ Certíssimo. Requisitos de usuário são de alto nível, linguagem simples, focados no comportamento externo.

✅ (2) Verdadeiro

Requisitos do produto… desempenho, memória, confiabilidade etc.

✔️ Correto. Isso descreve requisitos de produto (não funcionais).

✅ (3) Verdadeiro

Requisitos organizacionais… políticas, padrões, linguagem, entrega.

✔️ Perfeito. São restrições vindas da organização cliente/desenvolvedora.

✅ (4) Verdadeiro

Requisitos externos… interoperabilidade, legais, éticos.

✔️ Correto. São restrições fora da organização e do sistema.

❌ (5) Falso

Requisitos de domínio… “sem criar novos requisitos funcionais”

🚨 Aqui está a pegadinha.

Requisitos de domínio PODEM SIM gerar novos requisitos funcionais porque vêm das regras do negócio.

👉 Portanto, a afirmação está errada.

🧾 Gabarito final:

✅ V – V – V – V – F

Alternativa correta: V – V – V – V - F.

✔️ (1) Requisitos do usuário

Descrição em linguagem compreensível, alto nível, pode ser em língua natural. 👉 Verdadeiro — definição clássica (Sommerville).

✔️ (2) Requisitos do produto

Fala de desempenho, memória, confiabilidade, portabilidade, usabilidade. 👉 Isso é exatamente requisito de produto (não funcional). 👉 Verdadeiro

✔️ (3) Requisitos organizacionais

Políticas, padrões de processo, linguagem, método, entrega etc. 👉 Definição correta. 👉 Verdadeiro

✔️ (4) Requisitos externos

Interoperabilidade, legais, éticos. 👉 Também está correto. 👉 Verdadeiro

❌ (5) Requisitos de domínio

A frase diz que eles são “já existentes e sem criar novos requisitos funcionais”.

🚨 Aqui está o erro. 👉 Requisitos de domínio frequentemente geram novos requisitos funcionais, porque vêm das regras do negócio.

Exemplo rápido: Regra do banco → sistema deve calcular juros compostos ➡️ isso vira requisito funcional.

👉 Portanto é Falso.
